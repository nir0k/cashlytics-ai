# Pitfalls Research

**Domain:** Nodemailer SMTP + Token-Based Password Reset in Next.js 16 App Router + Auth.js v5
**Project:** Cashlytics v1.1 — Email & Password Reset
**Researched:** 2026-02-25 (expanded with deployment and integration pitfalls)
**Confidence:** HIGH (core security + Nodemailer patterns + Docker deployment) / MEDIUM (Auth.js v5 beta-specific behavior, session versioning)

---

## Critical Pitfalls

Mistakes that cause security breaches, broken flows, or silent failures that look correct during development.

---

### Pitfall 1: Nodemailer Transporter Created Per Request (Missing Singleton)

**What goes wrong:**
A new `nodemailer.createTransport()` call is made inside every Server Action or API handler that sends email. Each invocation opens and closes an SMTP connection, causing connection exhaustion under any load and significantly slower email sending (300–800ms of TLS handshake overhead per mail, paid again on every send).

**Why it happens:**
Server Actions run as serverless-style handlers. Developers write the transporter creation inline with the send call because it "works" in dev. Under Next.js App Router, each Server Action invocation is a new Node.js execution context — there is no HTTP-persist-alive for free, and a naive `createTransport` inside the function body gets called every single time.

**How to avoid:**
Create the transporter once as a module-level singleton. Next.js module caching means the same instance is reused for the lifetime of the process (including multiple requests in the same container):

```typescript
// src/lib/email/transporter.ts
import nodemailer from "nodemailer";

let transporter: nodemailer.Transporter | null = null;

export function getTransporter(): nodemailer.Transporter {
  if (!transporter) {
    transporter = nodemailer.createTransport({
      host: process.env.SMTP_HOST,
      port: Number(process.env.SMTP_PORT ?? 587),
      secure: process.env.SMTP_PORT === "465",
      auth: {
        user: process.env.SMTP_USER,
        pass: process.env.SMTP_PASS,
      },
    });
  }
  return transporter;
}
```

**Warning signs:**

- Email sending takes 400ms+ even for simple mails in production
- Docker logs show repeated TLS handshake events
- SMTP server shows hundreds of short-lived connections

**Phase to address:** SMTP Infrastructure Phase (Phase 1)

### Pitfall 2: Storing Raw Reset Token in Database

**What goes wrong:**
The plaintext token generated by `crypto.randomBytes()` is stored directly in the `password_reset_tokens` table. If the database is breached (SQL injection, backup leak, read replica exposure), an attacker can immediately use any unexpired token to reset any user's password — without brute force.

**Why it happens:**
Developers treat the reset token like a session value ("it's already random, why hash it?"). The parallel with password hashing is not obvious because the token is never "verified" like a password — it's compared directly from a URL parameter.

**How to avoid:**
Store `SHA-256(token)` in the database, send the raw token in the email URL. On validation, hash the incoming token and compare against the stored hash:

```typescript
import crypto from "crypto";

// Generate
const rawToken = crypto.randomBytes(32).toString("hex"); // 64 hex chars
const hashedToken = crypto.createHash("sha256").update(rawToken).digest("hex");

// Store hashedToken in DB, send rawToken in email URL

// Validate
const incomingHash = crypto.createHash("sha256").update(inputToken).digest("hex");
const record = await db.query.passwordResetTokens.findFirst({
  where: eq(passwordResetTokens.tokenHash, incomingHash),
});
```

**Warning signs:**

- `token` column in DB contains 64-char hex strings (raw, not hashed)
- Token column is directly compared to URL param: `where(eq(tokens.token, params.token))`
- No SHA-256 import in the token generation or validation code

**Phase to address:** Reset Token Phase (Phase 2)

### Pitfall 3: Timing Attack on Token Lookup

**What goes wrong:**
The token validation query returns "not found" immediately for invalid tokens but takes longer for valid-but-expired tokens (because the record is found first, then expiry checked in application code). This timing difference leaks information: an attacker can determine whether a token exists by measuring response time, which enables oracle attacks on token enumeration.

**Why it happens:**
Developers naturally write: "find the token, then check if it's expired" — two sequential steps. The database returns early on a miss, taking a different code path than a hit.

**How to avoid:**
Perform a single query that loads both record and expiry status, then use `timingSafeEqual` or a constant-time comparison for the final validation step. Alternatively, always execute the same database work regardless of outcome:

```typescript
import crypto from "crypto";

export async function validateResetToken(rawToken: string) {
  const incomingHash = crypto.createHash("sha256").update(rawToken).digest("hex");

  // Single query — always runs, result is null or record
  const record = await db.query.passwordResetTokens.findFirst({
    where: and(
      eq(passwordResetTokens.tokenHash, incomingHash),
      isNull(passwordResetTokens.usedAt),
      gt(passwordResetTokens.expiresAt, new Date())
    ),
  });

  // Return only boolean — no "not found" vs "expired" distinction to caller
  return record ?? null;
}
```

The consistent response pattern (same code path, single DB hit) eliminates the detectable timing differential.

**Warning signs:**

- Application code checks expiry AFTER finding the token: `if (token && token.expiresAt < new Date())`
- Different error messages returned for "token not found" vs "token expired" — these should be one message
- No single-query validation with all conditions combined

**Phase to address:** Reset Token Phase (Phase 2)

### Pitfall 4: Token Entropy Too Low

**What goes wrong:**
Using `Math.random()`, `Date.now().toString(36)`, `uuid()`, or `nanoid(6)` as a reset token. These either have predictable output (Math.random is not cryptographically secure), insufficient entropy for security (short nanoid), or deterministic-seeded entropy in some environments.

**Why it happens:**
These functions are familiar and feel "random enough" for display purposes. The security requirement for a password reset token is different: it must be unpredictable even to an attacker who can observe many tokens.

**How to avoid:**
Use `crypto.randomBytes(32)` — this gives 256 bits of cryptographically secure randomness. This is the Node.js built-in CSPRNG and is the correct primitive for security tokens:

```typescript
import crypto from "node:crypto";

// 32 bytes = 256 bits entropy = 64 hex chars
const token = crypto.randomBytes(32).toString("hex");
```

Do NOT use:

- `Math.random()` — not cryptographically secure
- `nanoid()` with short length — insufficient entropy under 128 bits
- `uuid()` (UUID v4) — only 122 bits, and version/variant bits are predictable

**Warning signs:**

- Token generation uses `Math.random`, `Date.now`, or `nanoid(n)` with n < 22
- Token is short enough to be human-readable (< 40 characters)
- `uuid` import in token generation without custom entropy source

**Phase to address:** Reset Token Phase (Phase 2)

### Pitfall 5: Token Not Invalidated After Single Use

**What goes wrong:**
A reset token remains valid after it has been used to reset a password. An attacker who intercepted the email link (via shoulder surfing, email forwarding rule, open shared inbox, etc.) can replay the URL hours or days later to set a new password and take over the account.

**Why it happens:**
The token is deleted or marked used only if the developer explicitly writes that step. It's easy to write the password update logic and ship it without the cleanup.

**How to avoid:**
Mark the token as used atomically with the password update. Use a database transaction so both operations succeed or both fail:

```typescript
await db.transaction(async (tx) => {
  // 1. Mark token used
  await tx
    .update(passwordResetTokens)
    .set({ usedAt: new Date() })
    .where(eq(passwordResetTokens.id, record.id));

  // 2. Update password
  const hashed = await hashPassword(newPassword);
  await tx.update(users).set({ password: hashed }).where(eq(users.id, record.userId));
});
```

Also consider invalidating ALL other pending reset tokens for the same user when one is used.

**Warning signs:**

- Token cleanup is a separate `await` call after the password update (not in a transaction)
- `usedAt` column exists in schema but never set in the action
- `DELETE` from tokens table happens before password update (reversed order — password update can fail and token is already gone)

**Phase to address:** Reset Token Phase (Phase 2)

### Pitfall 6: proxy.ts Does Not Allow /forgot-password and /reset-password

**What goes wrong:**
The `forgot-password` and `reset-password` pages are unauthenticated pages (by definition — users come here because they can't log in). But the existing `proxy.ts` only allows `/login` and `/register` as unauthenticated routes. Everything else redirects to `/login`. Result: clicking the reset link redirects to `/login` instead of the reset form, breaking the entire flow.

**Why it happens:**
The proxy.ts allowlist was written for the initial auth implementation. New public routes must be explicitly added — the default behavior is deny-all.

**How to avoid:**
Update the allowlist in `src/proxy.ts` in the same phase as the pages are created:

```typescript
// Current (proxy.ts)
if (pathname === "/login" || pathname === "/register") {

// Updated
if (
  pathname === "/login" ||
  pathname === "/register" ||
  pathname === "/forgot-password" ||
  pathname.startsWith("/reset-password") // handles /reset-password?token=...
) {
```

Also verify the `matcher` regex in `config` does not accidentally exclude these paths.

**Warning signs:**

- Visiting `/forgot-password` when logged out redirects to `/login`
- Reset email link opens `/login` instead of the reset form
- No mention of new public routes in proxy.ts when pages are added

**Phase to address:** Route + Page Phase (whichever phase creates the forgot-password pages)

### Pitfall 7: SMTP Credentials Missing from docker-compose.yml Environment Block

**What goes wrong:**
SMTP env vars are defined in `.env.example` and documented, but the `docker-compose.yml` `environment:` block does not forward them to the container. Nodemailer receives `undefined` for `SMTP_HOST` and silently falls back to a local SMTP connection to `localhost:25` (which doesn't exist in the container), causing all emails to fail with `ECONNREFUSED` or silently drop.

**Why it happens:**
The developer adds SMTP vars to `.env` and tests locally with `next dev` (where all env vars are auto-loaded). Docker Compose requires explicit var forwarding. The existing `docker-compose.yml` only forwards `DATABASE_URL`, `NEXT_PUBLIC_APP_URL`, and `AUTH_SECRET`.

**How to avoid:**
Add SMTP vars to the `environment:` block in `docker-compose.yml`:

```yaml
environment:
  - DATABASE_URL=${DATABASE_URL}
  - NEXT_PUBLIC_APP_URL=${NEXT_PUBLIC_APP_URL}
  - AUTH_SECRET=${AUTH_SECRET}
  # Email (v1.1)
  - SMTP_HOST=${SMTP_HOST}
  - SMTP_PORT=${SMTP_PORT}
  - SMTP_USER=${SMTP_USER}
  - SMTP_PASS=${SMTP_PASS}
  - SMTP_FROM=${SMTP_FROM}
```

Also update `Dockerfile` if it uses an `ENV` instruction list.

**Warning signs:**

- Emails work with `next dev` but fail in Docker
- `process.env.SMTP_HOST` logs as `undefined` inside the container
- Nodemailer error: `ECONNREFUSED 127.0.0.1:25` or `ECONNREFUSED localhost:587`
- docker-compose.yml environment block not updated alongside .env.example changes

**Phase to address:** SMTP Infrastructure Phase (Phase 1)

### Pitfall 8: HTML Email Rendered with Next.js/React Components (CSS-in-JS Not Supported)

**What goes wrong:**
Developer builds the email template as a React component with Tailwind CSS classes or CSS variables (`var(--font-syne)`, `from-amber-400/[0.15]`). These look fine in the browser preview. But email clients (Gmail, Outlook, Apple Mail) strip `<style>` tags, ignore CSS classes entirely, and do not process CSS variables. The resulting email is unstyled plain text.

**Why it happens:**
The Vault design system relies heavily on CSS custom properties, Tailwind utility classes, and `globals.css`. Developers assume email templates work the same way as web components. They do not — email HTML is effectively 2005-era HTML.

**How to avoid:**
Use inline styles for all email HTML. Either write raw HTML with inline `style=""` attributes, or use a purpose-built library like `@react-email/components` which handles inline style injection at render time:

```tsx
// react-email example — styles are inlined at render time
import { Html, Head, Body, Container, Text, Button } from "@react-email/components";

export function PasswordResetEmail({ resetUrl }: { resetUrl: string }) {
  return (
    <Html>
      <Body style={{ backgroundColor: "#08080a", fontFamily: "Arial, sans-serif" }}>
        <Container style={{ maxWidth: "600px", margin: "0 auto", padding: "40px 20px" }}>
          <Text style={{ color: "#f59e0b", fontSize: "24px", fontWeight: "bold" }}>Cashlytics</Text>
          <Button
            href={resetUrl}
            style={{ backgroundColor: "#f59e0b", color: "#08080a", padding: "12px 24px" }}
          >
            Passwort zurücksetzen
          </Button>
        </Container>
      </Body>
    </Html>
  );
}
```

Note: The Vault aesthetic (Syne font, glass cards) cannot be fully reproduced in email clients. Use a simplified version with websafe fallback fonts and flat colors.

**Warning signs:**

- Email template uses `className=` instead of `style=`
- Email CSS references `var(--font-syne)` or Tailwind utility classes
- No `@react-email/components` or equivalent inline-style library in `package.json`
- Email tested only in browser preview, not in an actual email client

**Phase to address:** Email Template Phase

### Pitfall 9: Email Enumeration via Password Reset Response

**What goes wrong:**
The `/forgot-password` form returns different responses for known vs unknown email addresses. For example: "If an account exists with this email, a reset link has been sent" vs "No account found with this email." The difference allows an attacker to enumerate which emails have Cashlytics accounts.

**Why it happens:**
It feels natural to give helpful feedback ("are you sure you typed the right email?"). The security implication is not immediately obvious.

**How to avoid:**
Always return the same response regardless of whether the email exists:

```typescript
// Server Action - always succeeds visually
export async function requestPasswordReset(email: string) {
  const user = await db.query.users.findFirst({
    where: eq(users.email, email.toLowerCase()),
  });

  // If user exists, send the email
  if (user) {
    await sendPasswordResetEmail(user);
  }

  // Always return success — never reveal whether email exists
  return { success: true, message: "Wenn dieses Konto existiert, wurde eine E-Mail gesendet." };
}
```

**Warning signs:**

- Server action returns different error messages for known vs unknown emails
- Response time differs between found/not-found (attacker can detect this via timing)
- Form shows "Email not found" as an error state

**Phase to address:** Reset Token Phase (Phase 2)

### Pitfall 10: Welcome Mail Blocks Registration Response

**What goes wrong:**
Sending the welcome email inside the `registerAction` Server Action with `await sendWelcomeEmail(user)` makes the registration response wait for the SMTP connection + TLS handshake + mail submission. If the SMTP server is slow or briefly unavailable, registration appears to hang or fails entirely — even though the account was created successfully.

**Why it happens:**
`await` is the natural way to call async functions. Making email sending synchronous with registration feels cleaner ("user should get the email before we redirect").

**How to avoid:**
Fire the welcome email without awaiting it. Accept that occasional SMTP failures won't block registration. Log failures for monitoring:

```typescript
// In registerAction
const user = await createUser(data);

// Non-blocking fire-and-forget
sendWelcomeEmail(user).catch((err) => {
  logger.error("Welcome email failed", { userId: user.id, error: err.message });
});

// Redirect immediately
redirect("/login");
```

Alternatively, use a queue (overkill for self-hosted scale) or a background task. For this project, fire-and-forget with logging is sufficient.

**Warning signs:**

- `await sendWelcomeEmail(user)` inside the registration Server Action
- SMTP timeout (30s default) can make registration appear frozen
- Registration succeeds in DB but user sees a timeout error

**Phase to address:** Welcome Email Phase

### Pitfall 11: Short Token Expiry Breaks Mobile Email Workflows

**What goes wrong:**
Reset token expiry is set to 5 or 10 minutes. Mobile users who open their email client on low connectivity, switch apps to check their password manager, then return to the email may find the link expired before they complete the flow.

**Why it happens:**
Developers optimize for security (shorter = safer) without considering real-world user workflow. 5 minutes is a common default copied from tutorials.

**How to avoid:**
Use a 1-hour expiry. This balances security (token is single-use anyway) with usability. The primary protection is single-use invalidation, not expiry time:

```typescript
const expiresAt = new Date(Date.now() + 60 * 60 * 1000); // 1 hour
```

Document the expiry duration in the email body: "This link expires in 1 hour."

**Warning signs:**

- Token expiry set to less than 30 minutes
- No expiry time mentioned in the email body
- User-facing error for expired tokens does not specify how long the link is valid

**Phase to address:** Reset Token Phase (Phase 2)

### Pitfall 12: Auth.js authVerificationTokens Table Conflict

**What goes wrong:**
Auth.js v5 with the Drizzle adapter includes an `authVerificationTokens` table in the schema (already present in this codebase at line 76 of `schema.ts`). If the password reset flow reuses this table or conflicts with it, the Drizzle adapter breaks — Auth.js v5 manages this table internally and expects a specific schema. Using it for custom password reset tokens corrupts the adapter's expected state.

**Why it happens:**
The table is named `auth_verification_tokens` and sounds like the right place to store verification/reset tokens. It is not — it is owned by Auth.js for magic link and email verification provider flows.

**How to avoid:**
Create a separate, dedicated table for password reset tokens. Do not reuse `authVerificationTokens`:

```typescript
// schema.ts — new table for v1.1
export const passwordResetTokens = pgTable("password_reset_tokens", {
  id: uuid("id").defaultRandom().primaryKey(),
  userId: uuid("user_id")
    .notNull()
    .references(() => users.id, { onDelete: "cascade" }),
  tokenHash: text("token_hash").notNull().unique(), // SHA-256 of raw token
  expiresAt: timestamp("expires_at", { mode: "date" }).notNull(),
  usedAt: timestamp("used_at", { mode: "date" }), // null = unused
  createdAt: timestamp("created_at").defaultNow().notNull(),
});
```

**Warning signs:**

- Password reset logic inserts into `auth_verification_tokens`
- Schema modifications to `auth_verification_tokens` beyond what Auth.js expects
- Auth.js adapter errors after password reset token cleanup operations

**Phase to address:** DB Migration Phase (must precede reset flow implementation)

### Pitfall 13: NEXT_PUBLIC_APP_URL Missing or Wrong in Reset Links

**What goes wrong:**
The password reset email contains a link built from `process.env.NEXT_PUBLIC_APP_URL`. If this env var is missing, wrong, or still set to `http://localhost:3000` in production, users receive broken links or links that point to the wrong host. The token works but the URL is unreachable.

**Why it happens:**
Developers set `NEXT_PUBLIC_APP_URL` during initial setup and forget to update it for production. Docker deployments may have different internal vs external URLs. The email template silently uses whatever value is present.

**How to avoid:**

1. Validate `NEXT_PUBLIC_APP_URL` at app startup — fail fast if missing or still localhost in production:

```typescript
// src/lib/email/config-validation.ts
export function validateEmailConfig() {
  const appUrl = process.env.NEXT_PUBLIC_APP_URL;
  if (!appUrl) {
    console.error("CRITICAL: NEXT_PUBLIC_APP_URL is not set. Email links will be broken.");
    return false;
  }
  if (process.env.NODE_ENV === "production" && appUrl.includes("localhost")) {
    console.error("CRITICAL: NEXT_PUBLIC_APP_URL still points to localhost in production.");
    return false;
  }
  return true;
}
```

2. Include a test link in the email that shows the actual URL being used:

```typescript
const resetUrl = `${process.env.NEXT_PUBLIC_APP_URL}/reset-password?token=${rawToken}`;
```

3. Add `NEXT_PUBLIC_APP_URL` to the startup health check in Docker.

**Warning signs:**

- Reset links contain `localhost:3000` when sent from production
- Users report "link doesn't work" but token is valid
- `NEXT_PUBLIC_APP_URL` not listed in docker-compose.yml environment block
- URL uses HTTP instead of HTTPS in production

**Phase to address:** SMTP Infrastructure Phase (Phase 1) — config validation

### Pitfall 14: Email Provider Rate Limiting Causes Silent Failures

**What goes wrong:**
Self-hosted users configure Gmail SMTP, a hosting provider's mail server, or a low-tier email service. These have strict rate limits (Gmail: ~500 emails/day for consumer accounts, many hosts: 50-100/hour). When password reset requests exceed limits, emails silently fail or get queued indefinitely. Users never receive reset links.

**Why it happens:**
Developers test with one or two emails and assume everything works. Rate limits aren't hit during development. Nodemailer's default behavior is to throw on failure, but if wrapped in a try/catch that just logs, the failure is invisible.

**How to avoid:**

1. Log all email send failures prominently with actionable messages:

```typescript
try {
  await transporter.sendMail(mailOptions);
} catch (error) {
  if (error.code === "EENVELOPE") {
    logger.error("SMTP rate limit hit. Check provider limits.", { error });
  } else {
    logger.error("Email send failed", { error: error.message });
  }
  throw error; // Re-throw to surface to caller
}
```

2. Document provider limits in `.env.example`:

```bash
# SMTP Configuration
# Note: Gmail consumer accounts have ~500 emails/day limit.
# For higher volume, use Gmail Workspace, Mailgun, SendGrid, etc.
SMTP_HOST=smtp.gmail.com
```

3. Add rate limiting on the forgot-password endpoint (3-5 requests per hour per email) to prevent users from accidentally exhausting provider limits by repeatedly requesting resets.

**Warning signs:**

- Email works in testing but fails intermittently in production
- Nodemailer error logs show `EENVELOPE` or rate limit messages
- Multiple password reset requests from same user without emails arriving
- Using Gmail consumer SMTP without app password

**Phase to address:** SMTP Infrastructure Phase (Phase 1) + Reset Token Phase (Phase 2)

### Pitfall 15: Missing SPF/DKIM Causes Email to Spam Folder

**What goes wrong:**
Emails are technically delivered but land in spam folders. Users don't see reset links and assume the system is broken. Self-hosted deployments using the server's local Postfix or a basic SMTP relay often lack proper email authentication (SPF, DKIM, DMARC records).

**Why it happens:**
SPF/DKIM/DMARC are DNS-level configurations outside the application code. Developers focus on the code working and don't realize email deliverability depends on DNS records they may not control.

**How to avoid:**

1. Document the deliverability requirements in README or deployment guide:

```markdown
## Email Deliverability Requirements

For reliable email delivery, ensure your domain has:

- SPF record: `v=spf1 include:_spf.google.com ~all` (or your mail server)
- DKIM signing enabled on your mail server
- DMARC record: `v=DMARC1; p=none; rua=mailto:postmaster@yourdomain.com`

Without these, emails may be marked as spam.
```

2. Use a `Reply-To` header with a real address for user support:

```typescript
const mailOptions = {
  from: process.env.SMTP_FROM,
  replyTo: "support@yourdomain.com",
  // ...
};
```

3. Include a plain-text version of the email — improves spam scores:

```typescript
const mailOptions = {
  html: htmlBody,
  text: "Reset your password at: " + resetUrl, // Plain text fallback
  // ...
};
```

4. Test with Mail Tester (mail-tester.com) before production launch.

**Warning signs:**

- Emails work but users report "never received"
- Email testing service shows low deliverability score
- Using a bare IP or unconfigured domain for SMTP
- No SPF/DKIM records in DNS for the FROM domain

**Phase to address:** Email Template Phase (Phase 4) — document deliverability requirements

### Pitfall 16: Password Reset Doesn't Invalidate Existing Sessions

**What goes wrong:**
A user resets their password via email (perhaps because they believe their account was compromised). The old session tokens remain valid — an attacker who stole the original session can continue to access the account even after the password is changed.

**Why it happens:**
JWT sessions (used in this codebase) are stateless. Changing the password doesn't invalidate existing tokens — they remain valid until their natural expiry. Developers naturally assume "password change = account secured" but this isn't true with JWT.

**How to avoid:**
After successful password reset, explicitly revoke all existing sessions. With Auth.js v5 and JWT strategy, this requires a session version/rotation approach:

```typescript
// After password update in resetPasswordAction
await db
  .update(users)
  .set({
    password: hashedPassword,
    // Add a sessionVersion field to force re-auth
    sessionVersion: sql`coalesce(session_version, 0) + 1`,
  })
  .where(eq(users.id, userId));
```

Then in the JWT callback, check the version:

```typescript
// In auth.ts callbacks
async jwt({ token, user }) {
  if (user?.sessionVersion !== undefined) {
    token.sessionVersion = user.sessionVersion;
  }
  // Invalidate if version mismatch
  if (token.sessionVersion !== tokenUser.sessionVersion) {
    return null; // Forces re-auth
  }
  return token;
}
```

**Simpler alternative for v1.1:**
Document that password reset does NOT invalidate sessions. Add a "Log out of all devices" button on the settings page as a future feature. For compromised accounts, the user should manually clear cookies.

**Warning signs:**

- No session invalidation logic after password change
- JWT strategy used (stateless by default)
- `sessionVersion` or similar field not in users table
- Password reset flow doesn't mention session security

**Phase to address:** Reset Token Phase (Phase 2) — document limitation or implement session rotation

### Pitfall 17: Development Email Testing Sends to Real Addresses

**What goes wrong:**
During development, developers test the password reset flow with their real email address. The reset email is actually sent to the real inbox. If the developer later uses the same code with test data that includes other people's emails, real users receive unintended test emails.

**Why it happens:**
There's no development-mode email interception. Nodemailer is configured with real SMTP credentials, and every `sendMail` goes to the real recipient.

**How to avoid:**

1. In development, override recipient to a test address:

```typescript
const isDev = process.env.NODE_ENV === "development";
const mailOptions = {
  to: isDev ? "dev-test@localhost" : recipientEmail,
  // ...
};
```

2. Or use a local mail catcher like Mailhog/Mailpit for development:

```bash
# docker-compose.dev.yml addition
services:
  mailpit:
    image: axllent/mailpit
    ports:
      - "1025:1025"  # SMTP
      - "8025:8025"  # Web UI
```

Then configure dev SMTP:

```bash
# .env.development
SMTP_HOST=localhost
SMTP_PORT=1025
SMTP_USER=
SMTP_PASS=
```

3. Add a prominent log in development mode:

```typescript
if (isDev) {
  console.log(`[DEV] Email would be sent to: ${recipientEmail}`);
  console.log(`[DEV] Reset URL: ${resetUrl}`);
}
```

**Warning signs:**

- Real emails sent during `next dev` testing
- No development mail catcher configured
- Test database contains real user email addresses
- `.env.development` has production SMTP credentials

**Phase to address:** SMTP Infrastructure Phase (Phase 1) — dev environment setup

## Technical Debt Patterns

| Shortcut                                 | Immediate Benefit     | Long-term Cost                               | When Acceptable      |
| ---------------------------------------- | --------------------- | -------------------------------------------- | -------------------- |
| Raw token in DB                          | Simpler query         | Full token exposure on DB breach             | Never                |
| `Math.random()` tokens                   | Familiar API          | Predictable tokens, trivially brute-forced   | Never                |
| Inline transporter creation              | Less abstraction      | Connection exhaustion, slow email            | Never                |
| Tailwind CSS in email                    | Consistent codebase   | Email clients strip class-based CSS entirely | Never                |
| Awaiting welcome email in register       | Clean sequential flow | Blocks registration on SMTP issues           | Never                |
| Single error message for found/not-found | Helpful UX            | Enables email enumeration                    | Never                |
| Reusing `auth_verification_tokens` table | Less migration work   | Breaks Auth.js adapter                       | Never                |
| Skipping SPF/DKIM setup                  | Faster deployment     | Emails land in spam                          | Never for production |
| Not invalidating sessions on reset       | Simpler flow          | Compromised sessions stay valid              | Document limitation  |
| Dev SMTP uses real addresses             | No extra setup        | Accidental emails to real users              | Never                |
| Hardcoded localhost in APP_URL           | Works locally         | Broken reset links in production             | Never                |

## Integration Gotchas

Common mistakes when integrating Nodemailer + Auth.js v5 + Drizzle in this specific stack.

| Integration                     | Common Mistake                               | Correct Approach                                               |
| ------------------------------- | -------------------------------------------- | -------------------------------------------------------------- |
| **Nodemailer + Server Actions** | Creating transporter per request             | Module-level singleton, export `getTransporter()`              |
| **Nodemailer + Docker**         | Env vars not forwarded in docker-compose.yml | Explicitly list all SMTP\_\* vars in environment block         |
| **SMTP_PORT=465**               | `secure: false` (default)                    | `secure: true` when port is 465, `false` for 587 (STARTTLS)    |
| **SMTP_PORT=587**               | Forgetting STARTTLS                          | Port 587 uses `secure: false` + `requireTLS: true`             |
| **Auth.js v5 + custom tokens**  | Using `authVerificationTokens` table         | Create separate `password_reset_tokens` table                  |
| **proxy.ts allowlist**          | Forgetting new public routes                 | Add `/forgot-password` and `/reset-password` to allowlist      |
| **Email templates**             | CSS variables and Tailwind classes           | Inline styles only — email clients do not process external CSS |
| **Next.js `redirect()`**        | Calling inside try/catch blocks              | `redirect()` throws internally — do not catch it               |
| **NEXT_PUBLIC_APP_URL**         | Left as localhost in production              | Validate at startup; fail if localhost in production env       |
| **Gmail SMTP**                  | Using account password                       | Use App Password (2FA required) or OAuth2                      |
| **Self-hosted Postfix**         | No authentication configured                 | Require SMTP AUTH; don't run as open relay                     |
| **Development email testing**   | Real SMTP credentials in dev                 | Use Mailpit/Mailhog locally; never send to real addresses      |

## Performance Traps

| Trap                                            | Symptoms                                            | Prevention                                                  | When It Breaks                  |
| ----------------------------------------------- | --------------------------------------------------- | ----------------------------------------------------------- | ------------------------------- |
| **Transporter-per-request**                     | 400ms+ email send times, SMTP connection exhaustion | Module-level singleton                                      | Any concurrent load             |
| **Token cleanup query on every request**        | DB load from scanning expired tokens                | Lazy cleanup (delete on use/lookup) or cron at app startup  | 10k+ reset requests accumulated |
| **Blocking registration on email send**         | Registration UI hangs on SMTP timeout               | Fire-and-forget with error logging                          | Any SMTP latency spike          |
| **Synchronous `renderToString` for email HTML** | Server Action slow response                         | Pre-render templates or use react-email's `render()` helper | High registration rate          |

## Security Mistakes

Domain-specific security issues for this email + password reset flow.

| Mistake                                  | Risk                                        | Prevention                                                           |
| ---------------------------------------- | ------------------------------------------- | -------------------------------------------------------------------- |
| **Raw token in DB**                      | Full account takeover on any DB read access | SHA-256 hash before storing                                          |
| **Insufficient token entropy**           | Brute-force in hours on powerful hardware   | `crypto.randomBytes(32)` minimum                                     |
| **Token reuse after password change**    | Account takeover via intercepted link       | Mark `usedAt` in same DB transaction as password update              |
| **Email enumeration**                    | Attacker discovers registered emails        | Always return identical response regardless of email existence       |
| **No rate limiting on /forgot-password** | SMTP cost abuse, user harassment            | Rate limit by IP + email: max 3 requests per hour per email          |
| **Reset token in URL fragment**          | Tokens visible in server logs               | Token in query string is fine; ensure server access logs are secured |
| **SMTP credentials in plaintext logs**   | Credential exposure                         | Never log SMTP_PASS; Nodemailer debug mode disabled in production    |
| **Timing oracle on token lookup**        | Token existence oracle attack               | Single-query validation combining all conditions                     |
| **Sessions survive password reset**      | Compromised sessions stay active            | Implement session versioning or document limitation explicitly       |
| **Missing SPF/DKIM/DMARC**               | Emails marked as spam, phishing risk        | Configure DNS records; use authenticated SMTP                        |
| **App URL still localhost in prod**      | Reset links point to wrong host             | Validate NEXT_PUBLIC_APP_URL at startup                              |

## UX Pitfalls

| Pitfall                                                         | User Impact                                   | Better Approach                                                                  |
| --------------------------------------------------------------- | --------------------------------------------- | -------------------------------------------------------------------------------- |
| **Expired link with no next step**                              | User stuck, can't recover                     | Error page should link back to /forgot-password                                  |
| **No expiry disclosed in email**                                | User delays action, link expires unexpectedly | State "This link expires in 1 hour" in email body                                |
| **"Token not found" vs "Token expired" message**                | Enables oracle attacks AND confuses users     | Use one message: "Link ungültig oder abgelaufen"                                 |
| **Password reset doesn't log user in**                          | User must log in again after reset — friction | Auto-login after successful reset OR clear redirect to /login with success toast |
| **HTML email with dark background breaks light-mode mail apps** | White text on white background                | Always include fallback text color, test in multiple clients                     |
| **No text/plain fallback in email**                             | Blocked by corporate spam filters             | Include both `html` and `text` fields in Nodemailer `sendMail` options           |
| **Reset email goes to spam**                                    | User never sees the link                      | Configure SPF/DKIM; test deliverability before launch                            |
| **No confirmation that email was sent**                         | User resends repeatedly                       | Show "Check your inbox" message with resend cooldown                             |
| **Reset link uses wrong domain**                                | Link doesn't work                             | Validate NEXT_PUBLIC_APP_URL; show actual URL in dev mode                        |

## "Looks Done But Isn't" Checklist

Things that appear complete but are missing critical pieces.

- [ ] **SMTP Setup:** `.env` vars defined but not forwarded in `docker-compose.yml` — verify `process.env.SMTP_HOST` inside container
- [ ] **Token Security:** Token column stores raw hex — verify SHA-256 hashing is applied before insert
- [ ] **Single-Use Enforcement:** Password updated but `usedAt` not set — verify token cleanup is in same transaction
- [ ] **proxy.ts Updated:** New pages exist but proxy still blocks them — verify `/forgot-password` and `/reset-password` are in allowlist
- [ ] **Email Enumeration:** Server action returns different messages for known/unknown email — verify identical response path
- [ ] **Email Rendering:** Template uses Tailwind classes — verify all styles are inline before testing in Gmail/Outlook
- [ ] **Text Fallback:** Email only has HTML part — verify `text` field is populated in sendMail options
- [ ] **Rate Limiting:** Forgot-password action has no rate limiting — verify IP/email-based limiting is implemented
- [ ] **Token Expiry in Email:** Email body does not state expiry duration — verify "expires in X" copy is present
- [ ] **Auth Table Isolation:** Using `auth_verification_tokens` for reset — verify a separate `password_reset_tokens` table is used
- [ ] **Transporter Singleton:** `createTransport` called inside Server Action function body — verify module-level singleton pattern
- [ ] **NEXT_PUBLIC_APP_URL:** Still set to localhost or missing — verify URL is correct for production environment
- [ ] **SPF/DKIM Records:** DNS not configured for email domain — verify with mail tester before launch
- [ ] **Session Security:** Password reset doesn't invalidate sessions — either implement session versioning or document limitation
- [ ] **Dev Email Safety:** Real SMTP in development sends to real addresses — verify dev uses mail catcher or test address
- [ ] **Email Provider Limits:** Gmail/hosting provider rate limits not documented — verify limits and implement client-side rate limiting

## Recovery Strategies

| Pitfall                          | Recovery Cost | Recovery Steps                                                                        |
| -------------------------------- | ------------- | ------------------------------------------------------------------------------------- |
| Raw token in DB                  | HIGH          | Invalidate all existing reset tokens immediately; rotate hash scheme in new migration |
| SMTP vars missing in Docker      | LOW           | Add to docker-compose.yml environment block, redeploy                                 |
| proxy.ts missing new routes      | LOW           | Add routes to allowlist, redeploy (no DB changes needed)                              |
